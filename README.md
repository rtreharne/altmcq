# Alternative MCQ Workshop App

Single-page Django facilitator app for the workshop **Rethinking MCQ Assessment**.

The app provides a slick facilitator screen, timed workshop sections, QR-code voting, live results, and local audio cues for the workshop flow.

## Setup

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
```

The scenario data is seeded by the Django migration.

## Run Locally

```bash
source .venv/bin/activate
python manage.py migrate
python manage.py runserver 0.0.0.0:8001
```

Open `http://127.0.0.1:8001/` on the facilitator machine.

If participants are on the same network, use the machine's local network address in place of `127.0.0.1`. The QR code points to the built-in `/vote/` form.

## Main URLs

- Facilitator page: `http://127.0.0.1:8001/`
- Participant vote form: `http://127.0.0.1:8001/vote/`
- Results JSON: `http://127.0.0.1:8001/results.json`

## Checks

```bash
python manage.py check
python manage.py test
```

## Scenario Handouts

Scenario Markdown files live in `scenarios/`. Each file includes a 5-criterion rubric scored from 1 to 5, for a total score out of 25.

Generate printable two-page PDFs with:

```bash
source .venv/bin/activate
python scripts/generate_scenario_pdfs.py
```

The PDFs are written to `scenario_pdfs/`.

The generator creates one PDF per scenario and a consolidated `scenario_pdfs/all-scenarios.pdf` containing all scenarios.

## Docker

Build and run the app as a single container:

```bash
docker build -t altmcq .
docker run --rm -p 8000:8000 \
  -e SECRET_KEY=local-dev-secret \
  -e DEBUG=False \
  -e SQLITE_PATH=/data/db.sqlite3 \
  -v "$PWD/.docker-data:/data" \
  altmcq
```

Open `http://127.0.0.1:8000/`.

The container runs migrations on start and serves Django with Gunicorn. Static files are collected during the image build and served by WhiteNoise.

## Deploy On Render

This repo is prepared for a single Render Docker web service on the free tier.

1. Push this repository to GitHub.
2. In Render, create a new Blueprint from the repo, or create a new Web Service and select Docker.
3. If using the included `render.yaml`, Render will create one free Docker web service.
4. The app stores SQLite at `/tmp/altmcq/db.sqlite3` on the free tier. This is enough for a live workshop session, but vote data may be lost when the service restarts or redeploys.
5. Render provides `RENDER_EXTERNAL_HOSTNAME`; Django uses it automatically for `ALLOWED_HOSTS` and CSRF trusted origins.

Required production environment variables:

- `SECRET_KEY`: generated by Render in `render.yaml`.
- `DEBUG`: `False`.
- `SQLITE_PATH`: `/tmp/altmcq/db.sqlite3`.

For persistent vote storage on Render, upgrade to a plan that supports disks and change `SQLITE_PATH` back to `/data/db.sqlite3`, then add a disk mounted at `/data`.

## Notes

- Runtime database state is stored in `db.sqlite3`, which is intentionally ignored by Git.
- The local virtual environment `.venv/` is ignored by Git.
- Workshop audio files are served from `workshop/static/workshop/audio/`.
